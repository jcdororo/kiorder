# 아키텍처

## 실시간 흐름
```
[테이블오더] → NestJS API → Prisma → Supabase DB
                                          ↓
                              Supabase Realtime (DB 변경 감지)
                                          ↓
                    [주방 화면] ←── 실시간 구독 ──── [테이블오더]
```

## 역할 분담
| 역할                        | 담당                |
|-----------------------------|---------------------|
| REST API + 비즈니스 로직    | NestJS              |

# 비즈니스 모델

## SaaS 구조
kiorder는 식당 사장님들이 구독하여 사용하는 SaaS 플랫폼이다.
구독 중인 사장님만 서비스를 이용할 수 있으며, 구독은 활성/비활성 2단계로 관리한다.
(티어제 플랜은 추후 고도화 예정)

## 역할 구조
| 역할 | 접근 조건 | 주요 권한 |
|------|-----------|-----------|
| 시스템 어드민 | 플랫폼 운영자 계정 | 사장님 목록 관리, 구독 승인/해제 |
| 사장님 (Store Owner) | 구독 활성 상태 | 매장 설정, 메뉴/주문/웨이팅 관리 |
| 손님 - Kiosk | 누구나 | 웨이팅 등록, 대기 현황 확인 |
| 손님 - Table Order | 누구나 | 메뉴 보기, 주문하기 |
| 주방 | 내부 이용 | 주문 목록 확인 및 상태 변경 |

# 화면 구조

## 시스템 어드민
- 사장님 목록 & 구독 관리 (활성/비활성 전환)

## 사장님 (관리자)
- 로그인 / 회원가입
- 대시보드 (매장 현황 한눈에)
- 테이블 배치 설정 ⭐ (드래그 & 드롭으로 자유 배치, 테이블별 좌석 수 설정)
- 주문 현황 ⭐ (설정한 테이블 레이아웃 UI 그대로 표시, 테이블 클릭 시 해당 주문 상세)
- 메뉴 관리
- 웨이팅 관리 (호출, 취소, 순서 조정)

**시나리오**
사장님은 드래그 & 드롭으로 실제 매장 구조와 동일하게 테이블을 배치하고 좌석 수를 설정한다.
주문 현황 화면은 설정한 레이아웃을 그대로 보여주며, 각 테이블을 클릭하면 해당 테이블의 주문을 확인할 수 있다.

## 키오스크 (손님)
- 웨이팅 등록
- 등록 완료 (대기번호 + QR코드 발급)
- 대기 현황 (실시간)

**시나리오**
고객은 키오스크에서 전화번호를 입력하고 대기표를 받는다.
전화번호를 입력하는 순간 자신의 대기번호가 몇번인지 확인 가능하고,
현재 몇번까지 입장하였는지 볼 수 있는 실시간 화면을 보여주는 QR코드가 생성된다.

## 테이블오더 (손님)
- 메뉴 목록 & 주문
- 주문내역 (실시간 상태 업데이트)

**시나리오**
고객은 자리에 앉아 테이블오더로 메뉴를 주문하고 주문내역을 볼 수 있다.
테이블은 URL 파라미터로 식별 (`/order?table=3`)

## 주방
- 주문표 목록 + 주문표 상세 (단일 페이지)
- 상태 변경 (접수 → 조리중 → 완료)

**시나리오**
주방장은 들어온 주문표를 하나씩 확인하고 상태를 변경한다.
새 주문은 Supabase Realtime으로 자동으로 나타난다.
